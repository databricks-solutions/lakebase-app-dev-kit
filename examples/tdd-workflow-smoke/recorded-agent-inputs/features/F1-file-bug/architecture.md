# Architecture: F1-file-bug

## Summary

Story S1-create-bug exercises a layered architecture: the E2E ACs (create-form-displays, form-accepts-title, form-accepts-description, form-submission-succeeds, redirect-to-detail-page) test the full stack from React UI through Express API to database persistence. The API AC (bug-created-with-open-status) isolates the backend business logic boundary. All ACs respect the inward dependency direction: UI depends on API, API depends on domain (Bug entity) and repository layer (BugRepository), domain depends on nothing.

## Layer Assignments

- **E2E (5 ACs):** create-form-displays, form-accepts-title, form-accepts-description, form-submission-succeeds, redirect-to-detail-page. Test the full application stack with a real UI, real API, and real database.
- **API (1 AC):** bug-created-with-open-status. Tests the POST /bugs endpoint and the application-layer service logic that creates a Bug entity with status='open'.

## Architectural Concerns Mapping

| Cross-Cutting Concern | Owner Layer | Implementation | Notes |
|---|---|---|---|
| Input Validation | API layer | BugController/validation middleware validates title and description (required, non-empty) before passing to service. Clear error messages name the offending field. | R2 enforces status validity at write time in domain layer. |
| Status Enforcement (R2) | Domain layer | Bug entity constructor rejects unrecognized status values. ORM constraint (CHECK) at database level. | Status is a recognized enum: 'open'. Never stored with invalid values. |
| Data Migration (R1) | Database layer + ORM | Alembic migrations use additive patterns: old columns preserved, new columns added with defaults. BugRepository uses ORM to abstract schema changes. | Older code paths continue to work during rollout. |
| Error Messages | API boundary | HTTP responses include field names and reason (e.g., `{"error": "Title is required"}`). | Preference from nfrs.md satisfied. |
| Transaction Integrity | Application layer | BugService.create() wraps bug insertion in a database transaction. All-or-nothing semantics for creating the Bug record. | Application layer owns transaction boundaries. |
| Logging | Application layer | BugService logs bug creation (ID, timestamp, status). Application instrumentation, not domain-aware. | Observability for debugging and audit trail. |

## Pattern Proposals

1. **BugCreateForm (React component, `/client/components/BugCreateForm`):** Encapsulates form state (title, description, loading, error) and event handlers. Single responsibility: capture user input and submit to API. No business logic.

2. **BugController (Express route handler, `/server/routes/bugs.js`):** POST /bugs endpoint. Delegates to BugService. Single responsibility: HTTP request/response handling, input validation, error translation to HTTP status codes.

3. **BugService (Application layer, `/server/services/BugService.js`):** `create(title, description)` method. Validates business rules (title non-empty, description non-empty), instantiates Bug entity with status='open', persists via BugRepository, returns created bug with ID. Single responsibility: application logic and transaction boundaries.

4. **Bug (Domain model, `/shared/models/Bug.ts`):** Entity with id, title, description, status, createdAt. Constructor enforces status is in recognized set. No knowledge of persistence or HTTP.

5. **BugRepository (Persistence layer, `/server/repositories/BugRepository.js`):** CRUD operations on Bug via ORM. All SQL generated by ORM; no raw SQL. Abstracts schema details from application layer.

## Risks

1. **Migration backward compatibility:** R1 depends on Alembic migrations following additive patterns. If a future iteration drops columns or renames tables without aliases, old code fails. Fitness function: automated migration test (create schema at N, run N+1, verify old queries still work).

2. **Status enum drift:** R2 enforces status validity, but the set of valid statuses may grow (e.g., 'closed', 'wontfix' in S2). If the domain enum is not kept in sync with the database CHECK constraint and API validation, a status can be stored that the application does not recognize. Fitness function: schema audit test (database CHECK constraint == domain enum set).

3. **UI-form-to-API contract:** BugCreateForm sends title and description via POST /bugs. If the API contract changes (field rename, new required field), the UI breaks silently. Fitness function: E2E test exercises the real form submission; API schema validation rejects malformed requests.

## Decisions

1. **Bug status on creation:** Always initialize to 'open'. Alternative: allow caller to specify status. Recommendation: keep it fixed for now (S1). R2 ensures no invalid status is stored. If future stories need parameterized status, the domain logic can be extended (S2 or later).

2. **Form validation locus:** Validation happens at the API boundary (BugController), not in the domain. Alternative: push validation into the domain (Bug constructor). Recommendation: API boundary is correct; domain should not depend on HTTP concerns. API layer translates HTTP validation errors into clear messages (nfrs.md preference). Domain enforces status validity only (R2).

3. **Redirect mechanism:** Client-side navigation (React Router) in BugCreateForm after API success. Alternative: server-side redirect (302). Recommendation: client-side is correct for SPA; stable URL contract (/bugs/{id}) is respected. Test strategy verifies the redirect with real E2E test.

## Test Strategy

Acceptance tests are **REAL integration tests** against the paired Lakebase branch database. No mocks, stubs, or in-memory substitutes for the data store.

For Python / pytest-bdd (the approach partner-asset-tracker uses):
- **Gherkin scenarios:** `.tdd/features/F1-file-bug/tests/features/create_bug.feature` (one feature file per story or feature).
- **Step definitions:** `.tdd/features/F1-file-bug/tests/step_defs/test_create_bug.py` (one step file per feature).
- **Fixtures + DB setup:** `.tdd/features/F1-file-bug/tests/conftest.py` (shared fixtures, Alembic migrations run once per test session via `alembic upgrade head`, FK-aware seed/teardown).
- **Test cleanup:** Each scenario cleans up with targeted, FK-aware DELETE statements keyed on the test's own data (e.g., DELETE FROM bugs WHERE id = ?). Never truncate.

**Verified via this real-DB behavior suite:**
- E2E ACs: create-form-displays, form-accepts-title, form-accepts-description, form-submission-succeeds, redirect-to-detail-page.
  - Exercise the full stack: browser loads app, renders BugCreateForm, captures input, submits to POST /bugs API, database persists Bug record, client navigates to /bugs/{id}.
  - Scenarios: "User creates a bug with title and description" (happy path), "User submits form with empty title" (error message), "User submits form with empty description" (error message), "User is redirected to bug detail after creation" (URL contract).
- API AC: bug-created-with-open-status.
  - Direct API call: POST /bugs with title and description. Assert database contains Bug with status='open'. Verify R2: invalid status rejection.
  - Scenarios: "Bug is created with open status", "Invalid status is rejected at write time".

**Migrations:** Before any test runs, `alembic upgrade head` applies all pending migrations to the paired Lakebase branch. Tests assume schema is current. Cleanup does not rebuild schema.
