# Orchestrated HIL intake for /design, /build, /ship (design, for review)

Status: **proposal, not yet implemented.** Defines a single orchestrated path
through `/design` -> `/build` -> `/ship` that runs identically for real and
test. The orchestrator interviews the human-in-the-loop (HIL) at intake steps,
generates the HIL-authored intake artifacts, and pauses at gates. In headless
test runs a **Human Proxy** stands in for the human. Adjacent to
(handoff contracts) (agent contracts) (per-role runtime).

## Principle: one path, the HIL is a human or a Human Proxy

There is **one orchestration path**, and the orchestrator drives it for both
real and test:

1. The orchestrator sequences the phases: `/design` -> `/build` -> `/ship`.
2. Within each phase it runs the intake interviews and stops at every HIL
   touchpoint: an **intake step** (an artifact must be authored) or a **gate**
   (an artifact must be approved).
3. At each touchpoint the orchestrator consults the **HIL**. The HIL has two
   implementations behind one interface:
   - **Real human** (interactive): answers the interview questions, edits the
     drafted artifact, approves the gate.
   - **Human Proxy** (headless test): a gate/step-aware stand-in. The
     orchestrator tells it the current phase + step + what is being asked; the
     proxy responds the way the human would, from pre-recorded material.

The test path is therefore **identical** to the real path. The only difference
is which HIL implementation is wired in. This replaces the old
"auto-approve / ci-mock-approver" framing (a bad name): the proxy is not an
auto-approver, it is a scripted human that knows the gates and steps.

### What the Human Proxy does at each touchpoint

The orchestrator hands the proxy the stage context (phase, step id, what it
needs). The proxy:

- **Intake step (artifact needed):** supplies the pre-recorded artifact for that
  step (e.g. the recorded `product-overview.md`, `nfrs.md`, `design-brief.md`,
  or an iteration's `feature-request.md`).
- **Gate (approval needed):** validates the expected elements are present and
  conformant (the diligent-reviewer behavior), then approves, recording the
  decision in the agent log exactly as a human's response would be recorded.
- It never invents intent and never skips a gate; if the recorded material for
  a step is missing or non-conformant, it refuses, the same as a hard-block.

The proxy's pre-recorded material is the set of intake md files the human
authored ahead of time (for the smoke: committed under the smoke's
`feature-requests/` + the recorded `product-overview.md` / `nfrs.md` /
`design-brief.md`).

## HIL-owned intake artifacts

Each is generated by an interview at the relevant phase and reviewed by the HIL
(human edits; proxy supplies the recorded version) before the consuming role
runs:

| Artifact | Generated during | Feeds | Conditional? |
|---|---|---|---|
| `product-overview.md` | `/design` intake | Spec Author (+ all roles) | always |
| `nfrs.md` | `/build` intake | Architect Reviewer | always |
| `design/design-brief.md` | `/build` intake | UX Designer | UI projects only |

`nfrs` = non-functional requirements; the artifact is just `nfrs.md`.
`design-brief.md` already exists in the kit (the UX Designer reads it); this
design specifies the interview that produces it.

## `/design` intake interview -> `product-overview.md`

The orchestrator runs this at the start of `/design`, before the Spec Author
phase. Question set (PO interview):
1. What is the product, in a sentence or two? Who uses it?
2. What do those users need to accomplish? (the core jobs)
3. What does the first usable version do, versus what comes later?
4. How do you expect the product to grow over the next few iterations?
5. What is explicitly out of scope / a non-goal?
6. Delivery expectations: what do you want to see after each sprint?

Output: `product-overview.md` (Product Owner voice; no implementation detail).
HIL reviews before the Spec Author reads it.

## `/build` intake interview -> `nfrs.md` (+ `design-brief.md`)

The orchestrator runs this when entering `/build`, before the Architect (and,
for UI projects, the UX Designer) flesh out their artifacts.

### Track A , NFR interview -> `nfrs.md` (feeds the Architect)

For each NFR category the HIL states a hard requirement, a preference, "N/A", or
"out of bounds":
1. Performance: latency / throughput targets?
2. Scalability: data + concurrency growth?
3. Security: authn/authz, data sensitivity, audit?
4. Observability: what must be logged / measurable?
5. Operability: deploy, config, runbook expectations?
6. Resilience: failure modes that must degrade gracefully?
7. Anything the architect should explicitly NOT pursue?

Output: `nfrs.md` with `## Required` / `## Preferences` / `## Out of bounds`.
The Architect must carry every `## Required` item into `architecture.json`
(coverage enforced). HIL reviews before the Architect runs.

### Track B , UX interview -> `design/design-brief.md` (feeds the UX Designer)

UI projects only. Follows the partner-asset-tracker approach: point at reference
websites and say what to take from each.
1. Is there a UI to design? (if no, skip this track)
2. Name 1-3 reference websites to model from.
3. For EACH reference, what specifically to take from it (brand, color, layout,
   tone)?
4. Hard brand constraints (logo, palette, font)?
5. Interaction / feedback expectations (e.g. no silent failures)?
6. Accessibility / device targets?

Output: `design/design-brief.md` with the required `## References` section
(each site + what to take) plus constraints. The UX Designer extracts tokens
into `design-guide.{md,json}` + `ia.md`, citing each reference. HIL reviews
before the UX Designer runs.

## /ship (orchestrated, design TBD)

`/ship` is the third orchestrated phase. It is in the sequence the orchestrator
drives and will have its own HIL touchpoints (e.g. release approval). Its
detailed interview/gates are out of scope for this doc and will be specified
separately; it is named here so the orchestration covers the full lifecycle.

## Conformance + the Human Proxy

- Conformance hard-blocks malformed artifacts at every gate,
  unchanged, for both human and proxy.
- New: when `nfrs.md` exists, the Architect's `architecture.json` must cover
  every `## Required` NFR (see Decision 1). The Human Proxy verifies coverage
  before approving the architecture gate, just as a human would.
- The proxy records each decision in the agent log as a `product-owner`
  (or relevant HIL role) gate event, so the relay is auditable and the test log
  matches what a real run would produce.

## Decisions (in flight, deciding each)

1. NFR coverage matching: explicit `brief_ref` on `architecture.json` NFR
   entries (auditable) vs. text association.
2. Uncovered `## Required` NFR: hard-block vs. warn-then-block-at-gate.
3. `nfrs.md` scope: project-level only vs. project-level + per-feature override.
